## 一、内存使用对比

### 1.1 不同上下文大小的内存占用

| 上下文 | F16 (MB) | Q4_0 (MB) | 节省 (MB) | 节省比例 |
|--------|----------|-----------|-----------|----------|
| 4,096 | 889 | 913 | -24 | -2.7% |
| 8,192 | 980 | 918 | **62** | **6.3%** |
| 16,384 | 1,013 | 927 | **86** | **8.5%** |
| 32,768 | 1,079 | 948 | **131** | **12.1%** |

**关键发现:**
- **上下文越大，TurboQuant 的内存优势越明显**
- 在 32K 上下文时，TurboQuant 节省 **131 MB (12.1%)** 内存
- 小上下文时差异不大，因为模型权重占主导

### 1.2 KV Cache 内存计算

```
KV Cache 内存 = 2 × layers × heads × dim × ctx × bytes_per_element

gemma3:1b: 26 layers × 8 heads × 256 dim

F16: bytes = 2
Q4_0: bytes = 0.5 (理论值，实际略有开销)
```

**理论节省:** 75% (实际测试 ~12%，因为模型权重不变)

## 七、结论

### 核心发现

1. **内存节省显著**: 大上下文时节省 10%+ 内存
2. **速度损失很小**: 仅 ~3-4%，可忽略
3. **上下文翻倍**: 相同内存支持 2 倍上下文
4. **质量接近 F16**: TurboQuant 4-bit 接近无损

### 最佳实践

**默认使用 Q4_0 (TurboQuant)**

```bash
# 推荐：平衡性能和质量
moxing ollama serve model --kv-cache q4_0

# 长上下文
moxing ollama serve model --kv-cache q4_0 -c 65536

# 质量优先
moxing ollama serve model --kv-cache f16
```
